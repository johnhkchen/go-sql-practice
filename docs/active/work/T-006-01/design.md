# Design: sync-sessions-collection (T-006-01)

## Requirements Summary

Create a PocketBase collection for sync sessions with:
- `progress` field (float, 0-1)
- `admin_token` field (text, for admin access)
- Public read access
- Write access restricted to requests with matching admin token
- Realtime subscription support (built-in)

## Design Options

### Option 1: Pure API Rules Approach
Use PocketBase's built-in API rules for all access control.

Pros:
- No custom code required
- Leverages PocketBase's native features

Cons:
- API rules can't validate request token against stored token
- Would expose admin_token in responses
- Not viable for token-based authentication

**Verdict**: Rejected - API rules can't handle token validation logic

### Option 2: Custom Routes Only
Bypass PocketBase's standard API entirely, use only custom routes.

Pros:
- Complete control over validation logic
- Can hide admin_token from responses

Cons:
- Loses automatic CRUD operations
- Must reimplement listing, viewing, updating
- More code to maintain

**Verdict**: Rejected - Unnecessary complexity for simple requirements

### Option 3: Hybrid Approach (Selected)
Use standard API for reads, custom route for updates.

Pros:
- Leverages PocketBase's built-in read operations
- Custom route handles token validation for updates
- Minimal custom code
- Maintains realtime subscription support

Cons:
- Two different endpoints for reads vs writes
- Slightly more complex API surface

**Verdict**: Selected - Best balance of simplicity and security

## Implementation Design

### Collection Schema

```go
sync_sessions collection:
- id: auto-generated by PocketBase
- progress: NumberField (float, min: 0, max: 1, default: 0)
- admin_token: TextField (required, min: 32 chars)
- created: auto-managed by PocketBase
- updated: auto-managed by PocketBase
```

### Token Generation Strategy

Use crypto/rand for secure token generation:
- 32 bytes of random data
- Encoded as hex string (64 characters)
- Generated server-side on session creation

Rationale:
- crypto/rand is cryptographically secure
- Hex encoding is URL-safe and simple
- 32 bytes provides 256 bits of entropy

### API Design

#### Standard PocketBase API (for reads):
- `GET /api/collections/sync_sessions/records` - List all sessions
- `GET /api/collections/sync_sessions/records/:id` - Get specific session
- Realtime: `/api/realtime` - Subscribe to changes

#### Custom Route (for writes):
- `POST /api/sync-sessions` - Create new session (generates token)
- `PATCH /api/sync-sessions/:id` - Update session (requires token)

Token validation:
- Passed as `X-Admin-Token` header
- Compared against stored `admin_token` field
- Returns 401 if missing or invalid

### Access Rules Configuration

Collection API rules:
- List: Allow all (empty rule)
- View: Allow all (empty rule)
- Create: Deny all (handled by custom route)
- Update: Deny all (handled by custom route)
- Delete: Deny all (not needed per requirements)

This ensures:
- Anyone can read sessions
- Only custom routes can modify data
- Admin token never exposed in standard API responses

### Routes Package Structure

Create new `routes/` package with:
- `routes/register.go` - Main registration function
- `routes/sync_sessions.go` - Sync session handlers

Functions:
- `Register(app)` - Called from main.go
- `createSyncSession()` - POST handler
- `updateSyncSession()` - PATCH handler

### Migration Integration

Add to existing `migrations/collections.go`:
- Check for sync_sessions collection
- Create if doesn't exist
- Set up fields with validation
- Configure API rules

## Security Considerations

### Token Security
- Tokens are visible in GET responses (acceptable per requirements)
- Tokens are never logged or displayed in errors
- Token comparison uses constant-time equality check

### Rate Limiting
- Not implemented (rely on PocketBase defaults)
- Could add if abuse detected

### Session Expiry
- Not implemented (not in requirements)
- Sessions persist indefinitely

## Error Handling

### Create Session Errors
- Database error: Return 500
- Validation error: Return 400

### Update Session Errors
- Session not found: Return 404
- Missing token header: Return 401
- Invalid token: Return 401
- Invalid progress value: Return 400
- Database error: Return 500

## Testing Strategy

### Unit Tests
- Token generation produces correct length
- Token validation logic

### Integration Tests
- Create session generates token
- Update with valid token succeeds
- Update with invalid token fails
- Standard API allows reads
- Realtime subscriptions receive updates

## Rejected Alternatives

### Alternative: JWT Tokens
Considered using JWT for admin tokens.

Rejected because:
- Overkill for simple authentication
- Adds complexity without benefit
- Random tokens are simpler and sufficient

### Alternative: Session Names/IDs
Considered adding human-readable session identifiers.

Rejected because:
- Not in requirements
- PocketBase ID is sufficient
- Can be added later if needed

### Alternative: Progress as Integer
Considered using integer (0-100) instead of float.

Rejected because:
- Float (0-1) is more standard
- Easier for percentage calculations
- Better precision for progress tracking

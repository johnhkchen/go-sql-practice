## Context

Two Go endpoint issues:

1. `routes/links_search_simple.go` line 82 uses `fmt.Sprintf` to embed user input directly into SQL, creating a SQL injection vulnerability. The `escapeLikePattern2` function escapes LIKE wildcards but doesn't prevent injection via the format string.

2. `routes/links_view.go` can't fetch the updated record after incrementing view_count because it references a `daos` package that doesn't exist in PocketBase v0.36.5. It returns `rows_affected` instead of the full updated record.

## Acceptance Criteria

- `links_search_simple.go` uses parameterized queries (bind variables) instead of string interpolation for all user input, OR the file is deleted if `links_search.go` already covers the same functionality
- `links_view.go` returns the full updated link record after incrementing view_count, using PocketBase v0.36.5's `app.FindRecordById()` or equivalent
- No SQL injection is possible on any endpoint
- Both endpoints return correct response shapes per their original ticket specs
